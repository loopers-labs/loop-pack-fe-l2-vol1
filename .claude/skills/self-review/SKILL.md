# /self-review 구현 내용 리뷰 스킬

구현한 결과물을 검증하는 스킬.

---

## 절차

1. `git diff`로 변경 파일 목록을 확인하고 변경된 모든 파일을 Read로 읽는다.
2. spec 파일과 CONVENTIONS.md 를 읽는다.
3. 아래 관점으로 검증하고 결과를 형식에 맞춰 보고한다.

---

## 검증 관점

### 1) spec 대비 기능 완성도

- spec의 완료 조건을 하나씩 대조
- 엣지 케이스 누락 (빈 상태 · 에러 상태 · 로딩 상태) 없는지 확인
- 완료 조건에 없지만 명백히 빠진 것 체크

### 2) 변경 내용 파악 및 영향 범위

- 파일별로 before & after가 무엇을 어떻게 바꿨는지 요약
- 그 변경이 spec 목적에 부합하는지
- 이 파일에 의존하는 다른 코드가 변경 때문에 깨지지 않는지 검증

### 3) 코드 품질

- 불필요한 코드, 데드 코드
- 하드코딩 값 (매직 넘버 · 인라인 문자열 · 색상/px)
- 불필요한 리렌더링, API / DB 불필요 호출
- 보안 문제
- 테스트 추가가 필요한 변경인지

### 4) 규칙 준수

- `pnpm build`(타입 체크) · `pnpm lint` 통과
- CONVENTIONS.md 준수
- AGENTS.md Boundaries 위반 없는지
- 기존 코드 패턴과 일관성

---

## 결과 형식

```
## Self Review 결과

**판정: PASS / FAIL** (blocker가 하나라도 있으면 FAIL)

### 기능 완성도
- [x] {완료 조건} — OK
- [ ] {완료 조건} — {빠진 점}

### 정적 검사
- `pnpm build` · `pnpm lint`: PASS / FAIL

### 지적 사항 (심각도순)
- `[blocker]` {파일:줄} — {문제} → {고칠 방법}
- `[개선]` {파일:줄} — {문제} → {고칠 방법}
- `[nit]` {파일:줄} — {문제} → {고칠 방법}
(없으면 "없음")
```

---

## FAIL 시

이슈를 수정한 뒤 self-review를 재실행한다. 최대 3회 반복, 그래도 FAIL이면 사용자에게 보고한다.
