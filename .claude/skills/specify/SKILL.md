# /specify 요구사항 정리

구현 전에 모호함을 줄이고 설계와 완료 조건을 명확히 하는 데 사용하는 스킬.
결과물은 `specs/{YYMMDD}-{name}.spec.md` 에 저장한다.

---

## 절차

### 1. 미러링

사용자의 요청을 듣고 이해한 내용을 정리하여 되묻는다.

```
"이렇게 이해했습니다:
- 목표: [목표]
- 포함: [범위]
- 제외: [비범위]
- 완료 조건: [언제 끝인지]
맞나요?"
```

**규칙**

- 사용자가 말하지 않은 것을 최소 1개 추론하여 빠진 게 없는지 확인
- 기술 선택은 여기서 하지 않고 인터뷰 단계에서
- 미러링은 최대 3회 시도하고 여전히 불명확하다면 직접 질문

사용자 확인 완료하면 spec 파일 생성

```bash
mkdir -p specs
```

```markdown
# {name} 스펙

## 목표

{사용자의 목표}

## 비범위

- {하지 않는 것}

## 확정 목표

{미러링 후 정제된 목표}
```

### 2. 코드베이스 조사

Glob/Grep/Read로 관련 기존 코드를 탐색한다.

- 이 기능과 관련된 기존 패턴
- 수정 대상 파일 구조
- AGENTS.md

결과를 spec 파일에 추가

```markdown
## 조사 결과

- {발견 1} (`file:line`)
- {발견 2} (`file:line`)
```

### 3. 인터뷰

조사 결과와 목표를 바탕으로 모호한 점을 시나리오 질문으로 끌어낸다.

**질문 규칙:**

- 추상적 질문 금지. "인증 어떻게 할까요?" ❌
- 시나리오 질문 지향. "토큰 만료 중 폼 제출하면 어떻게 해야 하나요?" ✅
- AskUserQuestion 도구 사용해 선택지 제공
- "에이전트가 판단" 선택지 항상 포함
- 2-3개 질문 후 결과 반영하고 추가 모호함이 있으면 반복

**종료 조건:**

- 핵심 동작, 범위, 에러/엣지 케이스가 명확해졌을 때
- 사용자가 "충분하다"고 할 때

결과를 spec 파일에 추가:

```markdown
## 결정 사항

- D1: {결정} — {근거}
- D2: {결정} — {근거}
```

### 4. 완료 조건과 태스크 도출

```markdown
## 완료 조건

- [ ] {측정 가능한 조건 1}
- [ ] {측정 가능한 조건 2}
- [ ] `pnpm build` · `pnpm lint` 통과

## 태스크

- T1: {할 일} — fulfills: 완료 조건 1
- T2: {할 일} — fulfills: 완료 조건 2
```

**완료 조건 규칙:**

- "잘 동작" 등 명확하지 않은 표현 지양 ❌ → "버튼 클릭 시 '성공했습니다' 토스트 표시" ✅
- 모든 조건이 검증 가능해야 함
- 타입 체크나 린트 통과는 항상 포함

AskUserQuestion으로 사용자 승인 확인

---

## spec 파일 위치

```
specs/{YYMMDD}-{name}.spec.md
```

`{YYMMDD}`는 작성일, `{name}`은 목표에서 kebab-case로 생성 (예: `specs/260618-login.spec.md`).

---

## 이 스킬을 쓰지 않아도 되는 경우

오타 수정, 사소한 스타일 변경 등 모호함이 없는 간단한 작업은 생략한다.
