## 컴포넌트 작성 규칙

- Props interface는 컴포넌트 파일 최상단에 정의
- 이벤트 핸들러 네이밍: Props는 `on{Event}`, 내부 함수는 `handle{Event}`
  - 예시) `<Component onClick={handleClick} />`
- 한 컴포넌트 안에서 두 가지 이상의 변경 이유가 포함되면 분리할 것(함께 바뀌지 않는 로직을 한 곳에 두지 않는다)
- 네이밍은 역할을 직관적으로 파악할 수 있도록 할 것
- 한 컴포넌트가 다른 컴포넌트들을 조합한다면 조합만, 내부에서 구현한다면 구현만 할 것
- 서로 무관한 상태가 한 곳에 있을 경우 분리할 것. 그 상태를 쓰는 UI와 함께 별도로 떼어낸다.
  - 예시) 목록 정렬 상태 + 모달 열림 상태
- props의 개수는 최대한 적게, 많다면 연관된 상태를 객체로 묶는 것을 고려할 것
- boolean props가 많다면 유니온 타입 단언이나 단일 enum으로 컴포넌트 내부에서 불가능한 상태가 표현되지 않도록 할 것
- props로 계산되는 값을 state에 복사하지 말 것. 파생 가능한 값은 렌더 중에 계산한다.
- 외양 확장은 prop의 개수를 늘리지 말고, 위임(스타일 주입 및 다형성)으로 해결한다.
- Context를 도입하기 전에 children으로 끌어올리는 것을 우선 고려한다.

## 코드 리뷰 및 품질 규칙

- default export 사용 금지
- 의도가 이름에 명확히 드러나도록 네이밍할 것 (data, temp, flag, handleClick1 등 금지)
- 한 함수/컴포넌트는 한 가지 일만 하도록 작성할 것
- 파생 가능한 값은 useState + useEffect로 동기화하지 말고 직접 계산할 것
- 컴포넌트 내부에 직접 fetch를 사용하지 말고 API 레이어를 분리할 것
- 커밋 메시지는 반드시 `Conventional Commits` 타입을 준수할 것 (feat:, fix:, chore:, refactor:)
