## 💡 제1원칙: 설명 가능성 & 책임 모델

- **설명할 수 없는 코드는 커밋 금지**: AI 코드가 버그를 내도 최종 책임은 1차 작성자(인간)에게 있음.
- **책임 분리**: 도구(Lint/Format)는 컨벤션·타입만 보장하며, 정확한 비즈니스 로직 판별은 인간이 수행함.

## 🛠️ 핵심 코드 품질 기준 (ESLint 강제 불가 항목만)

1. **의도적 네이밍**: `data`, `temp`, `flag`, `value` 등 모호한 이름 절대 금지. (예: `flag` ➡️ `isAdult`)
2. **단일 책임**: 한 함수/컴포넌트는 한 가지 일만 수행. 이름에 'and'가 필요하면 분리할 것.
3. **파생 값 계산**: `useState`+`useEffect`를 통한 상태 동기화 금지. 렌더링 시점에 직접 계산.
4. **Early Return**: 조건부 렌더링은 early return으로 처리하되, 모든 hook 호출부 아래에 배치.
5. **기존 코드 재사용**: 새 유틸/컴포넌트 작성 전, 반드시 `src/` 내 기존 코드를 먼저 검색하고 재사용할 것.

## 🚫 하네스(방어벽)

ESLint가 강제할 수 있는 룰은 `eslint.config.js`에서 관리한다. 우회 수단(`eslint-disable`, `ts-ignore`, `--no-verify`)은 사용 금지.

## 🚀 Git & PR 규칙

- **브랜치**: `main` 기준 생성. `feature/weekN-기능명` 또는 `fix/weekN-버그명` 형식 준수.
- **커밋 게이트**: husky(`lint-staged`) 통과 필수. `--no-verify` 강제 우회 금지.
- **커밋 메시지**: `type: 명확한 동사 + 작업 대상` (예: `feat: 로그인 기능 구현`)
- **PR 형식**: 제목에 `[volume-n]` 명시. 본문에 핵심 리뷰 포인트 3개 이내 작성 및 하단 문구 필수 포함:
  > `"○○는 AI로 생성 후 직접 검토·수정했습니다."`
