# 접근성 규칙

UI를 변경할 때는 기능 동작뿐 아니라 키보드, 스크린 리더, 시각적 인지, 보안 속성을 함께 확인한다. `eslint-plugin-jsx-a11y`가 정적 문제를 잡지만, 실제 접근성은 수동 확인이 필요하다.

## 기본 원칙

- 의미에 맞는 HTML 요소를 사용한다.
- 클릭 가능한 요소는 기본적으로 `button` 또는 `a`를 사용한다.
- `div`/`span`에 `onClick`을 붙여 버튼처럼 만들지 않는다.
- 모든 인터랙션은 키보드만으로 접근 가능해야 한다.
- focus outline을 제거하지 않는다. 커스텀 focus 스타일은 `:focus-visible`을 제공한다.

## 텍스트와 구조

- 페이지나 섹션은 heading 계층을 논리적으로 유지한다.
- 시각적 제목과 스크린 리더 제목이 어긋나지 않게 한다.
- 아이콘만 있는 버튼에는 접근 가능한 이름을 제공한다. 예: `aria-label`.
- form control은 label과 연결한다.
- 에러 메시지는 어떤 입력과 관련 있는지 알 수 있어야 한다.

## 이미지와 아이콘

- 의미 있는 이미지는 내용을 설명하는 `alt`를 제공한다.
- 장식 이미지는 `alt=""`로 둔다.
- 장식 SVG는 `aria-hidden="true"`와 `role="presentation"`을 사용한다.
- 텍스트 이미지는 피하고 실제 텍스트로 표현한다.

## 링크와 버튼

- 버튼은 `type="button"`, `type="submit"` 등 의도를 명시한다.
- 새 탭 링크(`target="_blank"`)는 `rel="noreferrer"` 또는 `rel="noopener noreferrer"`를 함께 둔다.
- 링크 텍스트는 목적지를 설명해야 한다. `여기`, `click`처럼 맥락 없는 문구를 피한다.

## 상태와 피드백

- 로딩, 에러, 빈 상태를 사용자에게 명확히 보여준다.
- 비동기 동작 중 중복 제출이 가능하면 disabled 상태나 pending 표시를 제공한다.
- 중요한 상태 변화는 필요한 경우 ARIA live region을 검토한다.
- 색상만으로 상태를 구분하지 않는다.

## 시각 디자인

- 텍스트와 배경의 색 대비를 확인한다.
- hover에만 의존하지 않는다. keyboard focus와 touch 환경을 함께 고려한다.
- 애니메이션이나 motion은 과하지 않게 사용하고, 필요한 경우 `prefers-reduced-motion`을 고려한다.
- 반응형에서 글자 크기, 터치 타겟, 레이아웃 순서가 유지되는지 확인한다.

## 수동 체크리스트

UI 변경 후 최소한 다음을 확인한다.

- Tab / Shift+Tab으로 주요 인터랙션에 도달할 수 있는가?
- 현재 focus 위치가 시각적으로 보이는가?
- Enter / Space로 버튼 동작이 가능한가?
- form label과 에러 메시지가 연결되어 있는가?
- 스크린 리더가 불필요한 장식 요소를 읽지 않는가?
- 모바일 폭에서 내용이 잘리거나 순서가 이상해지지 않는가?
