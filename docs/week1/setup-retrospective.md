# 1주차 환경 세팅 회고 — 단계별 기록

> **목적**: "왜 이렇게 했는가"를 박제한다. 잘 간 길뿐 아니라 **잘못 가던 길·병목·AI와 내가 실패한 지점**까지 남긴다.
> 과제 제1원칙(*설명할 수 없는 코드는 커밋하지 않는다*)의 연장 — 설명을 문서로 박아둔다.
>
> **최종 목표**: 모든 모호함을 해소(=결정 완료)했을 때, 환경 세팅 전체가 깔끔한 **1개의 atomic 커밋**(`chore: 환경 세팅`)이 된다.
> 시행착오는 이 문서에 남고, git 히스토리에는 결론만 1줄로 남는다.
>
> **읽는 법** — 각 단계를 다섯 렌즈로 본다:
> ✅ 정상 진행 · 🚧 병목 · ❌ 실패(AI / 나) · ⚖️ 트레이드오프 · ✔️ 결정

## 진행 상태 (한눈에)

| 단계 | 항목 | 상태 |
| --- | --- | --- |
| 0 | 접근 방식 정하기 | ✔️ 결정 |
| 1 | CLAUDE.md (AI 행동 규칙서) | ✔️ 초안 |
| 2 | ESLint v10 flat config | ✔️ 작성·동작 검증 |
| 3 | Prettier | ✔️ 작성·검증 |
| 4 | husky + lint-staged 커밋 게이트 | ✔️ 설정 |
| 5 | tsconfig strict 보강 | ✔️ 적용·검증 |
| 6 | 게이트 동작 검증 (고의 lint 오류) | ✔️ 차단 확인 |
| 7 | 1 커밋으로 정리 | ✔️ 완료 |

## 단계 0 — 접근 방식부터 부러뜨리다

- **✅ 정상 진행**: 환경 세팅 코드를 짜기 전에 "어떻게 협업할지"를 먼저 합의했다.
- **❌ AI의 실패**: Claude가 ESLint/Prettier/husky 설정을 *한 번에 자동 생성*하려 했다 → 사용자가 중단.
- **❌ 나의 실패(될 뻔)**: 그대로 뒀으면 "왜 이 룰인지" 모른 채 통과시킬 뻔했다 — 과제 금물(*디폴트/복붙 수용*) 위반.
- **🚧 병목**: AI가 "효율"을 이유로 앞서나가는 성향 ↔ 과제가 요구하는 "단계별 직접 판단".
- **⚖️ 트레이드오프**: 속도(한 번에 끝) ↔ 학습·통제력(단계별 승인). → **후자 채택** (과제 목적이 판별력 훈련이므로).
- **✔️ 결정**: `CLAUDE.md`에 *"증강 코딩 — AI는 제안만, 승인 후 수행"* 규칙을 박아 재발을 막았다.

## 단계 1 — CLAUDE.md: AI의 행동 규칙서

- **✅ 정상 진행**: 제1원칙 + 기술 스택 + 컴포넌트/리뷰/커밋 규칙 + Never Do를 한 파일에 정리.
- **⚖️ 트레이드오프**:
  - *분량* — 모든 규칙을 길게 ↔ 핵심만 짧게. → "지웠을 때 AI가 실수하면 남긴다" 기준으로 **짧게**.
  - *커밋 co-author 서명* — 도구 기본 관행(Claude 서명 추가) ↔ 본 레포 금지. → **금지** (히스토리를 사람 기준으로 유지).
- **❓ 남은 모호함**: 없음.

## 단계 2 — ESLint v10 flat config

- **✅ 정상 진행**: 핵심 룰만 직접 선별해 `eslint.config.mjs` 작성, **stdin 위반 코드로 8개 룰 동작 검증** 완료
  (`any`/`console`/`==`/`no-empty`/미사용 변수/중첩 삼항이 전부 error로 잡힘).
- **✔️ 부러뜨린 모호함 3개**:
  - `exhaustive-deps` → **warn** 채택(발제 기본). *왜 error가 아닌가*: false positive 여지가 있고,
    잘못된 강제는 더 위험한 우회(룰 통째 disable)를 부른다. 대신 PR에서 **0 warning**으로 관리.
  - `no-console` → **error 전면 금지**. *왜 예외(warn/error 콘솔)를 안 두나*: 과제 본질이 "잔재 0".
    의도적 로깅이 필요해지면 그때 팀 합의로 `allow`를 연다.
  - 룰 구성 → **recommended 펼침 대신 핵심 룰 직접 선별 + 근거 주석**. 과제 금물(통째 복붙) 회피.
- **🎁 뜻밖의 수확**: `js.recommended`의 `no-empty`가 `catch (e) {}`(에러 삼키기)까지 잡아준다 —
  과제 "좋은 코드 기준"의 *에러를 명시적으로 처리한다*와 정확히 맞물림.
- **❓ 남은 모호함**: 없음.

## 단계 3 — Prettier

- **✅ 정상 진행**: 기존 스타일을 그대로 코드화(`semi:false`·`singleQuote`·2-space·`trailingComma:all`).
  `prettier --check` 결과 **src 코드는 이미 전부 통과**(diff 0) → 판단이 맞았다. ESLint와도 충돌 없음(`eslint-config-prettier`).
- **🚧 예상 못 한 병목**: 진짜 모호함은 "포맷 규칙"이 아니라 **"적용 범위"**였다.
  `prettier --check .`가 멘토 발제 원본·도구 상태 파일(`.claude`/`.omc`)·마크다운까지 싹 잡으며 드러남.
- **✔️ 부러뜨린 모호함**: **"코드만 관리"** 채택.
  - 도구 상태(`.claude`/`.omc`)·`pnpm-lock.yaml`·에셋 → ignore (우리 코드가 아님).
  - 마크다운(`*.md`) → ignore. *왜*: 문서는 산문(표·의도적 줄바꿈)이라 Prettier가 의도를 해치고,
    멘토가 제공한 발제 원본을 재포맷하면 "내가 안 쓴 diff"가 생긴다.
- **⚖️ 트레이드오프**: 레포 전체 통일(일관성) ↔ 코드만(원본 보존·작은 책임 범위). → 후자.
- **❓ 남은 모호함**: 없음.

## 단계 4 — husky + lint-staged 커밋 게이트

- **✅ 정상 진행**: `husky init`으로 pre-commit 훅 생성, 내용을 `pnpm exec lint-staged`로 교체.
  lint-staged 대상 — `*.{ts,tsx}`: `eslint --fix`+`prettier --write`, `*.{css,json,mjs}`: `prettier --write`. **`*.md` 제외**(단계 3의 "코드만 관리"와 일관).
- **✔️ 부러뜨린 모호함**: 게이트는 **error만 차단, warn은 통과**(`--max-warnings` 미설정).
  *왜*: exhaustive-deps를 warn으로 둔 취지(false positive·정당한 빈 deps)를 게이트에서 뒤집지 않기 위해.
  warn은 신호로 보이되 커밋을 막지 않는다.
- **⚖️ 트레이드오프**: 0-warning 강제(엄격) ↔ warn 통과(warn 선택과 일관). → 후자.

## 단계 6 — 게이트 동작 검증 (먼저 검증함)

- **✅ 검증 완료**: `any` + `console`(둘 다 `--fix`로 못 고침)을 가진 임시 파일을 staged 후 커밋 시도
  → lint-staged의 `eslint --fix`가 **error 2건으로 실패 → 커밋 차단(exit 1)**, HEAD 불변.
- **🎯 핵심**: 이것이 과제가 요구하는 *"고의 lint 오류로 게이트가 실제로 막는지"* 검증.
  `--no-verify` 우회는 CLAUDE.md에서 금지(규칙). 검증 후 임시 파일은 정리했다.

## 단계 5 — tsconfig strict 보강

- **✅ 정상 진행**: `strict: true` 추가 후 `tsc -b` → **기존 코드 전부 통과**(깨지는 곳 없음). 안전하게 보강.
- **✔️ 부러뜨린 모호함**: 의도적 누락(함정)으로 보고 **켠다**. *왜*: `no-explicit-any`(ESLint)는
  명시적 `any`만 잡지만, strict의 `noImplicitAny`/`strictNullChecks`가 *암묵적* 구멍까지 막아 1차 하네스를 완성한다.
- **🎁 뜻밖의 수확**: `package.json`을 손으로 편집했더니 Prettier 규칙과 어긋났고 `format`이 잡아줬다 —
  **설정 파일조차 사람 손이 닿으면 어긋난다 → 기계에 맡긴다**는 1차 하네스 철학의 실증.
- **❓ 남은 모호함**: 없음.

## 단계 7 — 1 커밋으로 정리

- **✅ 정상 진행**: 모호함을 전부 부러뜨린 뒤, 환경 세팅이 **단일 atomic 커밋**(`chore: ... 1차 하네스 구축`)으로 떨어졌다.
- **✔️ 부러뜨린 모호함**:
  - 커밋 구성 → **환경 세팅 1커밋 + 문서 1커밋**. *왜*: 설정과 문서는 다른 논리적 변경이라 분리(atomic).
  - `.omc/` → **`.gitignore`에 추가**(개인 도구 상태, 커밋 대상 아님).
- **🔁 자기참조 검증**: 이 커밋 자체가 방금 만든 게이트(pre-commit lint-staged)를 통과해야 들어갔다 —
  "AI가 만든 커밋에도 똑같이 걸린다"가 실제로 작동.

## 🎯 종합 회고 — 무엇이 정상 진행됐고, 무엇이 병목이었고, 누가 어디서 실패했나

### ✅ 정상 진행된 것

- 도구 전부 최신(ESLint v10·TS strict·Prettier·husky)으로 **충돌 없이 설치·동작**, 전 단계 그린(lint/format/typecheck/build).
- **게이트가 실제로 깨진 코드를 차단**함을 고의 오류로 검증(단계 6).
- 기존 코드 스타일을 코드화해 **불필요한 diff 0**.
- 단계마다 모호함을 **명시적으로 부러뜨리고 근거를 남김** → 결과가 깔끔한 1커밋.

### 🚧 병목이었던 것

- **AI가 효율을 이유로 앞서나가려는 성향**(단계 0). 과제가 요구하는 "단계별 직접 판단"과 충돌.
- **표면 문제 ≠ 진짜 문제**(단계 3). "포맷 규칙"인 줄 알았던 모호함의 본질은 "적용 범위"였다.
- **사람 손이 닿으면 어긋난다**(단계 5). 수동 편집한 `package.json`이 기계 규칙과 불일치.

### ❌ AI와 내가 실패한 부분

- **AI**: 환경 세팅을 한 번에 자동 생성하려다 과제 취지(직접 판단)를 건너뛸 뻔 → 중단당함. → `CLAUDE.md` "증강 코딩" 규칙으로 교정.
- **AI**: 커밋에 co-author 서명을 넣는 기본 관행 → 사용자가 명시적으로 금지 → 규칙화.
- **개발자(나)**: `tsconfig` strict 누락을 처음엔 못 보고 지나칠 뻔 → 회고를 쓰는 과정에서 발견·보강.
- **공통 교훈**: 모호함을 "나중에"로 미루면 커밋이 지저분해진다. **단계마다 부러뜨려야 1커밋이 깨끗하다.**

### 다음 주차로 가져갈 것

- 게이트(1차 하네스)가 섰으니, 이제 게이트가 **잡지 못하는 것**(파생 상태·관심사 분리·네이밍 의도)은
  사람 리뷰의 몫 — R2(500줄 컴포넌트 분리)에서 본격적으로 다룬다.

## 단계 3~7 — (진행하며 채운다)
