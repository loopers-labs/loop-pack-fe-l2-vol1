## ESLint: "어떤 룰을 넣을지"가 진짜 질문이었다

### 배경

이번 주 환경 세팅의 핵심은 ESLint flat config를 **직접** 설계하는 것이었다.
스캐폴딩 기본값이나 airbnb 프리셋을 그대로 쓰는 게 금지된 조건이라, 룰 하나하나를 선택해야 했다.

처음엔 "넣을 수 있는 좋은 룰"을 찾는 방향으로 접근했는데, 금방 막혔다.
ESLint 룰은 수백 개고, "좋아 보이는 것"을 다 넣으면 1차 하네스가 아니라 소음이 된다.

### 결국 쓴 선택 기준

1. **자주 발생하는 패턴인가** — 팀/AI가 실제로 자주 쓰는 코드 패턴을 잡는지
2. **자명하게 나쁜 패턴인가** — "왜 이게 나쁜가?"를 설명하기 쉬운 것만

이 두 조건을 동시에 만족하는 룰만 넣는 것으로 범위를 잡았다.

### 실제로 넣은 룰과 이유

| 룰                                  | 수준  | 이유                                                                               |
| ----------------------------------- | ----- | ---------------------------------------------------------------------------------- |
| `rules-of-hooks`                    | error | Hooks 호출 순서는 React의 근간 — 위반하면 런타임 오류                              |
| `exhaustive-deps`                   | warn  | stale closure 문제를 잡되, 의도적 누락 케이스가 있어 warn으로 타협                 |
| `consistent-type-assertions: never` | error | `as XXX` 타입 단언은 TypeScript를 우회하는 것 — AI가 자주 씀                       |
| `no-restricted-syntax` (handleXXX)  | error | `onXXX`는 props 전달용 예약, 핸들러는 `handleXXX` — 쉽게 지킬 수 있어 error        |
| `id-denylist` / `id-length`         | warn  | `data`, `temp`, 1글자 이름은 의도가 드러나지 않음 — 단, 카운터(`i`,`j`,`k`)는 예외 |

### `exhaustive-deps`를 warn으로 둔 이유

처음엔 error로 설정했다가 warn으로 내렸다.
이유: **의존성을 의도적으로 누락하는 케이스가 실제로 존재한다.**
(마운트 시 1회만 실행하는 effect 등)

이 룰을 error로 막으면 그 케이스마다 `eslint-disable`을 쓰게 되고,
그게 쌓이면 오히려 더 중요한 경고를 묻어버린다.
warn으로 두고 **스스로 판단하게 하는 것**이 이 룰의 올바른 사용이라고 결론냈다.

## CLAUDE.md: 방치에서 설계로

### 문제

이전까지 CLAUDE.md를 만들어도 금방 방치했다.
규칙이 쌓이면 길어지고, 길어지면 AI가 읽는 건지 모르겠고, 결국 안 씀.

### 이번 주 발견한 것

과제 가이드에서 CLAUDE.md 작성 원칙을 접했다:

> "이 줄을 지우면 AI가 실수하는가?" — 아니면 삭제.

이게 핵심이었다. CLAUDE.md는 팀 규칙의 전체 문서가 아니라,
**AI가 없으면 틀릴 가능성이 있는 것만 적는 곳**이다.

또 하나: **구조 분리**가 가능하다.
스택 정보 / 컴포넌트 규칙 / 리뷰 체크리스트처럼 카테고리로 나누면
수정할 때 어디를 건드려야 할지 명확해지고, 관리 비용이 확 줄었다.

### 다음 할 일

실무 프로젝트용 CLAUDE.md를 새로 설계해보려 한다.
기준은 이번에 배운 것 그대로 —

- 짧게 유지 (설명 안 되는 줄은 삭제)
- 카테고리 분리 (스택 / 규칙 / 리뷰 흐름)
- 주기적으로 pruning

## 한 줄 요약

> ESLint는 "좋은 룰 모음"이 아니라 "팀이 자주 실수하는 패턴을 막는 기계"다.
> CLAUDE.md도 마찬가지 — 길면 죽는다, 짧고 틀릴 때 읽히는 것만 남긴다.
