# 1주차 — 코드 리뷰 & AI 협업 환경 구축

> "잘 돌아가는 코드"와 "좋은 코드"는 다르다.
> 이번 주는 기능 구현이 아니라, **그 차이를 구분하는 눈**과 **그걸 강제하는 환경**을 만든다.

## 왜 이 과제를 하는가

- **AI 시대의 병목은 "쓰는 속도"가 아니라 "읽고 판별하는 능력"이다.** AI는 명백히 틀린 코드보다 _그럴듯하게 틀린_ 코드를 쏟아낸다(개발자 불만 1위). 그래서 구현 스킬보다 **판별력**을 먼저 훈련한다.
- **제1원칙 — 본인이 설명할 수 없는 코드는 커밋하지 않는다.** AI가 쓴 코드도 머지하는 순간 내 코드다. "AI가 짜줬는데요"는 변명이 안 된다. 이 10주 내내 적용되는 단 하나의 규칙이고, 나머지는 전부 이 원칙을 지키기 위한 기술이다.
- **책임 모델 — 누가 무엇을 책임지나**

  | 역할      | 책임지는 것                                          |
  | --------- | ---------------------------------------------------- |
  | 👤 작성자 | 정확성·버그·동작의 **1차 책임 전부**                 |
  | 🔍 리뷰어 | 설계 조언·리스크 식별·지식 공유 (버그 부재 보장은 ✗) |
  | 🤖 기계   | 컨벤션·포맷·타입·테스트 통과                         |

- **1차 하네스(lint + type)를 먼저 세우는 이유** — ① 결정적이다(같은 코드 → 항상 같은 결과), ② 가장 싼 시점(작성 중·커밋)에 돌아 버그 비용을 최소화, ③ AI가 스스로 고치며 수렴하는 검증 루프의 기반이다. **1차 하네스가 약한 레포에서는 AI 에이전트도 약해진다.**

## 이번 주 과제

| #   | 과제                                                               | 완료 기준                                                 |
| --- | ------------------------------------------------------------------ | --------------------------------------------------------- |
| 1   | **환경 세팅** — ESLint(flat config) + Prettier + husky + CLAUDE.md | 고의 lint 오류 커밋으로 **게이트 동작을 검증**한 커밋 1개 |
| 2   | **과제 구현 + 셀프 리뷰** — 4단 체크리스트 적용                    | PR 본문에 **AI 생성 부분 표기** 포함                      |

## 어떤 포인트로 진행하나

### ① 환경 — 머리로 배운 기준을 기계에 새긴다

- ESLint flat config + `react-hooks` recommended(`rules-of-hooks`·`exhaustive-deps` 포함). 핵심 규칙은 **`error` 레벨로, 우회 불가능하게**. (`warn`은 쌓이면 아무도 안 본다.)
- husky + lint-staged로 **커밋 게이트** 구성 — lint 미통과 = 커밋 불가. 설정 후 _일부러 깨진 파일을 커밋해 실제로 막히는지_ 검증한다. `--no-verify` 우회 금지.
- `CLAUDE.md`(타 에이전트는 `AGENTS.md`)로 팀 규칙을 AI에 주입. 원칙은 **짧게** — "이 줄을 지우면 AI가 실수하는가?" 아니면 삭제.
- git hook은 **AI가 만든 커밋에도 똑같이** 걸린다. 이게 "AI를 통제한다"의 가장 구체적인 형태.

### ② 셀프 리뷰 — 리뷰 요청 전, 내 diff를 처음부터 끝까지 직접 읽는다

1. **동작** — happy path 말고 로딩/에러/빈 상태 + **상태 전이**(있다→없다, 성공→실패)까지 머릿속에서 돌려봤나
2. **의존성** — import한 패키지가 실제 존재? 기존 유틸 두고 중복 생성 안 했나? hook 의존성 배열 완전한가?
3. **타입·린트 침묵** — 새 `any`/`as`/`@ts-ignore`/`eslint-disable`이 없나 (있다면 전부 "기계가 경고하려던 자리")
4. **표면적** — `dangerouslySetInnerHTML`, 입력 무검증 렌더링 같은 XSS 구멍 없나

→ 그리고 PR 본문에 한 줄: _"○○는 AI로 생성 후 직접 검토·수정했습니다."_ (리뷰어에게 어디를 주의 깊게 볼지 알려주는 예의)

## 🚫 금물 — 이렇게는 하지 마세요

특히 ① 환경 세팅에서, 아래는 **제1원칙("설명할 수 없는 코드는 커밋하지 않는다")** 위반입니다.

- **프로젝트가 깔아주는 디폴트 설정을 그대로 쓰지 말 것.** 스캐폴딩 도구가 넣어주는 기본 ESLint/설정을 검토 없이 수용하면, "왜 이 룰인지" 모른 채 통과시키는 것. (이 레포 베이스에 ESLint를 일부러 안 넣은 이유다 — 직접 세워라.)
- **기존에 쓰던 방식을 맹신하지 말 것.** `eslint-config-airbnb` 같은 익숙한 프리셋을 "원래 이거 쓰니까"로 복붙하지 않는다. 익숙함은 정당성이 아니다.
- **룰을 통째로 복붙하지 말 것.** ESLint를 **1차 하네스의 관점**에서 보고 — _"이 룰이 그럴듯하게 틀린 코드를 잡는가? `error`로 강제할 가치가 있는가?"_ — **어떤 룰을 수용/거부할지 스스로 결정**하고, 그 근거를 설명할 수 있어야 한다.

## 좋은 코드 기준 (10주 내내 쓰는 기준표)

- 의도가 이름에 드러난다 (`data`/`temp`/`flag` ✗)
- 한 함수/컴포넌트가 한 가지 일을 한다
- **파생 가능한 값은 계산한다** — `useState` + `useEffect`로 동기화하지 않는다 _(최중요 패턴)_
- 타입이 코드를 설명한다 (`any`로 회피 ✗)
- 조건부 렌더링은 early return으로 (단, **모든 hook 호출 뒤에서**)
- 에러를 명시적으로 처리한다 (`catch(e) {}` ✗)
- 기존 코드를 재사용한다 (비슷한 유틸 매번 새로 생성 = AI 고질병)

---

> 막히면: AI에게 **"왜?"를 먼저** 물어라 (설명 요청은 학습, 코드 위임은 학습이 아니다). 그래도 안 풀리면 "어떤 상황에서 뭘 시도했는지"와 함께 슬랙에 질문.
