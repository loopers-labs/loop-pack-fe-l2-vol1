# Week 02 — 리팩토링 과제 Spec

## 주제

주문/결제 페이지의 Bad Code Smell 판별 → 리팩토링 → 근거 → 기록

## 프로세스

1. Bad Code Smell 판별
2. 리팩토링
3. 근거 기록
4. 셀프 리뷰 4단계 통과 → 결과 기록
5. PR에 AI 생성 부분 표기, 리팩토링 부분마다 한 줄 근거

## 11전략 체크리스트

### 경계

| #   | 전략          | 설명                                        |
| --- | ------------- | ------------------------------------------- |
| ①   | 변화의 경계   | 변경 이유가 다른 코드가 한 곳에 섞여 있는가 |
| ②   | 구현 vs 조합  | 세부 구현과 조합 로직이 분리되어 있는가     |
| ③   | God Component | 한 컴포넌트가 너무 많은 책임을 지고 있는가  |
| ④   | 성급한 추상화 | 불필요한 추상화가 도입되었는가              |

### 계약

| #   | 전략                          | 설명                                               |
| --- | ----------------------------- | -------------------------------------------------- |
| ⑤   | props는 적게, 이름은 역할대로 | props가 과다하거나 이름이 역할을 드러내지 않는가   |
| ⑥   | boolean 폭발 → enum           | 여러 boolean props가 하나의 상태를 표현하고 있는가 |
| ⑦   | 파생 상태 + key               | 파생 가능한 값을 별도 state로 관리하고 있는가      |
| ⑧   | 확장은 위임으로               | 확장 시 기존 컴포넌트를 수정해야 하는 구조인가     |

### 합성

| #   | 전략                     | 설명                                                          |
| --- | ------------------------ | ------------------------------------------------------------- |
| ⑨   | Context 전에 composition | Context 없이 composition으로 해결 가능한가                    |
| ⑩   | children vs slot         | children과 slot 패턴이 적절히 사용되고 있는가                 |
| ⑪   | Drilling vs Context      | prop drilling이 과도하거나, Context가 반사적으로 도입되었는가 |

## 금물

- 냄새가 아닌데 "고쳤다" 하지 말 것 — 필요 없는 추상화, 과한 상태 끌어올리기가 대표적. 과잉 리팩토링도 제1원칙 위반.
- 강의 예제 해법을 그대로 복붙하지 말 것 — 슬라이드 코드를 옮기면 전이가 아니라 암기.
- 동작을 바꾸지 말 것 — 가려졌던 버그가 드러나 고쳐지는 것은 예외.
- 타입·린트를 off하는 식으로 우회하지 말 것.
